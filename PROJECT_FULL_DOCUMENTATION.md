# Pothole Detection System - Full Updated Documentation

## 1. Project Overview

This project detects potholes using deep learning and computer vision.

Current recommended pipeline:
- Train a PyTorch CNN on a binary dataset (Pothole vs Plain).
- Run realtime webcam or video-file inference using the trained `.pth` model.
- Use optional multi-pothole bounding-box tracking overlay in realtime/video mode.

Classes:
- `Pothole` -> class `1`
- `Plain` -> class `0`

Important:
- The model was originally a classifier.
- Realtime bounding boxes are generated by image-processing region proposals + patch classification + tracking (not full detector training like YOLO).

## 4. Main Pipelines

## 4.1 PyTorch Training (Recommended)

File:
- `backend/pytorch/train_gpu_pytorch.py`

What it does:
1. Loads train and test data from `backend/my_dataset`.
2. Applies grayscale + resize to 300x300.
3. Uses augmentation on train set (flip/rotation).
4. Trains CNN with weighted loss to reduce class bias.
5. Evaluates on test split every epoch.
6. Saves best model as `backend/pytorch/pytorch_pothole_gpu_model.pth`.

Notes:
- Designed for CUDA-enabled PyTorch.
- This is the current practical GPU path on this setup.

## 4.2 TensorFlow Pipelines (Legacy/Optional)

These scripts are preserved and path-updated, but TensorFlow GPU availability depends on platform/runtime setup.

## 5. Realtime and Video Inference (PyTorch)

File:
- `backend/pytorch/pytorch_realtime_video_predictor.py`

Supports:
1. Webcam/mobile camera inference.
2. Video-file inference with optional saved output video.
3. Camera listing and interactive camera selection.
4. Multi-pothole tracking with IDs and confidence per track.
5. Scene modes:
- `road` (stable, conservative)
- `screen` (sensitive for monitor/image-grid testing)

### 5.1 Overlay Meaning

On screen:
- `PLAIN` / `POTHOLE`: frame-level status from active tracked detections.
- `Conf: XX%`: strongest active pothole track confidence.
- `Frame: N`: current frame number processed.
- `FPS: Y`: processing/display frame rate.
- `Pothole #k ZZ%`: tracked pothole ID and confidence for that box.

### 5.2 How Multi-Pothole Detection Works

Per frame:
1. Generate candidate dark/irregular regions using morphological operations.
2. Filter bad candidates by geometry + texture checks.
3. Classify each candidate patch with trained PyTorch model.
4. Keep high-confidence patches.
5. Apply NMS to remove overlapping duplicates.
6. Track detections over time by IoU matching and smoothing.

This provides practical multi-box tracking but is still not equivalent to a fully supervised object detector trained with bounding-box labels.

## 6. Commands (Current)

Run from project root.

### 6.1 Environment

```powershell
python -m venv .venv
.\.venv\Scripts\Activate.ps1
python -m pip install --upgrade pip
pip install numpy opencv-python scikit-learn imutils
pip install --upgrade --force-reinstall torch torchvision torchaudio --index-url https://download.pytorch.org/whl/cu128
```

### 6.2 Train PyTorch Model

```powershell

```

Longer run:

```powershell
$env:EPOCHS='100'
python backend/pytorch/train_gpu_pytorch.py
```

### 6.3 List Available Cameras

```powershell
python backend/pytorch/pytorch_realtime_video_predictor.py --list-cameras --max-camera-id 10
```

### 6.4 Webcam Inference

Interactive camera choice:

```powershell
python backend/pytorch/pytorch_realtime_video_predictor.py --source webcam
```

Fixed camera id:

```powershell
python backend/pytorch/pytorch_realtime_video_predictor.py --source webcam --camera-id 1 --threshold 0.85
```

### 6.5 Video File Inference

```powershell
python backend/pytorch/pytorch_realtime_video_predictor.py --source video --video-path "D:/path/input.mp4" --output-path "outputs/pothole_pred.mp4" --show --threshold 0.85
```

### 6.6 Screen/Collage Test Mode

Use when camera sees monitor with many pothole photos:

```powershell
python backend/pytorch/pytorch_realtime_video_predictor.py --source webcam --scene-mode screen --threshold 0.70
```

## 7. Accuracy and Stability Notes

Why results can fluctuate:
- Small test set causes unstable metrics.
- Domain shift (real road vs phone/monitor images).
- Lighting, blur, camera motion affect region proposals.

Useful tuning:
- For fewer false positives: increase `--threshold` (for example 0.88).
- For more sensitivity: lower `--threshold` (for example 0.70).
- For monitor/grid tests use `--scene-mode screen`.

---

## 8. Key Limitations

1. Bounding boxes are heuristic + patch classification, not detector-trained boxes.
2. Realtime performance depends on camera quality and GPU/CPU load.
3. Generalization depends strongly on dataset quality and diversity.
4. TensorFlow GPU path can be environment-limited.
